# Ревью проекта `activate`

Дата: 2026-03-17

## Краткий итог

Проект в рабочем состоянии, но есть несколько важных зон риска: криптографический fallback для паролей, подавление ошибок миграций БД, и «шумные» артефакты в корне репозитория. Ниже — приоритезированный список.

## Приоритетные находки

### 1) [High] Слабый fallback-хеш пароля при отсутствии `passlib`
- В `cabinet_system/security.py` при недоступном `passlib` используется один `sha256` c солью (`sha256$salt$digest`).
- Это существенно слабее memory-hard алгоритмов (Argon2) и упрощает оффлайн-перебор при компрометации базы.
- Риск особенно высок, если окружение запускается без `passlib`/`argon2`.

**Рекомендация:**
1. Сделать `passlib[argon2]` обязательной зависимостью (без fallback в production).
2. Если fallback нужен только для dev/test — явно блокировать его в production через проверку `ENVIRONMENT`.

---

### 2) [Medium] Подавление ошибок миграций БД
- В `app.py` блок `try/except Exception: pass` вокруг `ALTER TABLE ...` глушит любые ошибки.
- Это затрудняет диагностику «частично применённых» схем и может приводить к скрытым багам на проде.

**Рекомендация:**
1. Логировать исключение с контекстом SQL-команды.
2. Использовать нормальный миграционный инструмент (Alembic) вместо best-effort DDL на старте.

---

### 3) [Medium] Потенциальная путаница с конфигурацией и секретами
- `app.py` принудительно читает `bot.env` из каталога приложения.
- Такой подход может приводить к дрейфу конфигурации между средами и риску случайного хранения секретов рядом с кодом.

**Рекомендация:**
1. Стандартизировать источник конфигурации (переменные окружения + секрет-хранилище).
2. Оставить `.env` только для локальной разработки, исключая боевые значения из репозитория.

---

### 4) [Low] В корне репозитория есть подозрительные/мусорные файлы
- В корне присутствуют файлы вида `-H`, `POST`, `x-secret:`, `user-agent:` и т.п.
- Похоже на артефакты ручных `curl`-запусков; они засоряют проект и могут случайно содержать чувствительные данные.

**Рекомендация:**
1. Удалить артефакты из Git.
2. Добавить точечные правила в `.gitignore` (или вынести такие эксперименты в `scratch/` вне репозитория).

## Что улучшить дальше (roadmap)

1. Ввести централизованные миграции (Alembic) и health-check на целостность схемы.
2. Добавить SAST/linters в CI (ruff/flake8, bandit, pip-audit).
3. Зафиксировать security-baseline: парольная политика, обязательный Argon2, ротация ключей, журналирование security-событий.
4. Провести cleanup корня репозитория и зафиксировать соглашение по структуре проекта.
